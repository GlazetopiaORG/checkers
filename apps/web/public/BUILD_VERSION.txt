KEY FIX in 5.0.12 — root cause of "{"moves":[]}":

The previous commitSession() implementation did NOT verify that its UPDATE
matched any rows. When `.eq('status', 'pending')` race-safety check
silently filtered the UPDATE (RLS, race, etc.), the function returned a
fabricated "active" view to the client while the DB row stayed at 'pending'.
Then every subsequent getLegalMoves() call short-circuited because
`row.status !== 'active'`, returning empty arrays.

Fix: commitSession now uses .select() on the UPDATE to confirm the row
was actually changed. If zero rows match, it reloads the row; if the
reload still shows pending, it throws INTERNAL_ERROR with a pointer
to investigate RLS.

Other 5.0.12 changes:
  - getLegalMoves logs row status/turn whenever it returns []
    (so Vercel function logs show exactly why moves are empty)
  - onSquareClick now gates on view.status/view.turn (server-authoritative)
    not on `phase` (which can lag)
  - Visible debug panel from 5.0.11 still present (top-right corner)
